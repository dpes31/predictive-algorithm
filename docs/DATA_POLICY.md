# Data Policy

상태: Gate 0 승인 정책  
적용 범위: 과거 당첨결과, 신규 회차, 예측결과, 평가결과

## 1. 단일 원본 원칙

과거 당첨번호는 하나의 원본 데이터셋에서만 관리합니다.

초기 HTML MVP:

```text
data/draws.json
```

Supabase 이전 후:

```text
lottery_draws 원본 테이블
```

HTML, JavaScript, 통계차트, 테스트 파일은 원본을 읽어 파생 데이터를 생성해야 하며 당첨번호를 별도로 복사해 관리하지 않습니다.

## 2. 필수 필드

```text
draw_no              회차
draw_date            추첨일
numbers               오름차순 당첨번호 6개
bonus_number          보너스번호
source                원본 출처
source_reference      출처 식별정보
verification_status   검증 상태
verified_at           공식 대조 완료시각
checksum              레코드 체크섬
created_at            최초 생성시각
updated_at            마지막 변경시각
locked                 잠금 여부
```

## 3. 검증 상태

```text
pending_manual
→ auto_checked
→ verified
→ locked
```

- `pending_manual`: 사용자 또는 관리자가 입력했으나 공식 독립 대조 전
- `auto_checked`: 자동 형식검사와 보조출처 비교 완료
- `verified`: 공식출처와 대조 완료
- `locked`: 예측과 아카이브에 사용하는 확정 레코드

공식 대조가 완료되지 않은 레코드는 예측 엔진 입력에서 제외합니다.

## 4. 무결성 검사

각 레코드는 다음 조건을 만족해야 합니다.

- 회차는 양의 정수
- 회차 중복 없음
- 당첨번호는 정확히 6개
- 당첨번호는 서로 다름
- 모든 번호는 1~45
- 번호는 오름차순으로 정규화
- 보너스번호는 1~45
- 보너스번호는 본번호와 중복되지 않음
- 추첨일 형식은 ISO `YYYY-MM-DD`
- 인접 회차 날짜가 일반적으로 7일 간격인지 검사
- 데이터셋에 결측 회차가 없는지 검사
- 전체 파일 및 레코드 체크섬 확인

날짜·회차의 예외가 발견되면 자동 수정하지 않고 검토 대상으로 표시합니다.

## 5. 정정 정책

검증된 데이터를 직접 덮어쓰지 않습니다.

정정 절차:

1. 기존 값 보존
2. 정정 사유 기록
3. 사용한 공식 근거 기록
4. 변경 전·후 체크섬 저장
5. 영향받는 예측·평가 결과 식별
6. 새 데이터 버전 생성

과거 예측은 당시 사용한 데이터 버전에 연결된 상태로 보존합니다.

## 6. 1230회 상태

사용자 제공값:

```text
회차: 1230
추첨일: 2026-06-27
당첨번호: 3, 8, 9, 22, 28, 42
보너스번호: 45
```

현재 정책상 공식 독립 대조 완료 전에는 `pending_manual`입니다. 공식 대조 후 `verified` 및 `locked`로 승격합니다.

## 7. 예측결과 데이터

각 예측 실행은 다음 정보를 보존합니다.

```text
target_draw_no
input_last_draw_no
data_version
model_version
model_weights
candidate_sets[5]
uncertainty_summary
created_at
locked_at
prediction_hash
status
```

예측 잠금 후 후보 번호를 수정하지 않습니다. 오류가 발견되면 기존 실행을 폐기 표시하고 새로운 모델 또는 데이터 버전으로 별도 실행을 생성합니다.

## 8. 아카이브 표시 정책

아카이브는 다음을 표시합니다.

- 1회부터 최신 검증 회차
- 회차와 추첨일
- 당첨번호 6개
- 보너스번호
- 검증상태
- 출처
- 해당 회차의 사전 잠금 예측이 존재하는 경우 후보 5세트
- 후보별 실제 일치번호
- 당시 모델·데이터 버전

번호 색상:

- 1~10: 노랑
- 11~20: 파랑
- 21~30: 빨강
- 31~40: 회색
- 41~45: 초록

## 9. 향후 Supabase 이전

Gate 4 승인 후 다음 항목을 이전합니다.

- 원본 당첨번호 테이블
- 데이터 변경 이력
- 모델 버전
- 잠금 예측 실행
- 후보 조합
- 평가결과

이전 전후 레코드 수, 체크섬, 첫 회차·마지막 회차를 비교해 동일성을 검증합니다.
