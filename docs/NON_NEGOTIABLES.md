# Non-Negotiables

이 문서는 프로젝트에서 임의로 바꿀 수 없는 핵심 원칙을 요약합니다.

## 알고리즘

1. M0 균등 무작위모형을 제거하지 않는다.
2. M1 지속, M2 반전, M3 구조변화 가설을 동시에 유지한다.
3. 단일 출현빈도 점수식으로 축소하지 않는다.
4. 무작위성 검정은 비균등모형 활성화 게이트로 사용한다.
5. 신호가 없거나 재현되지 않으면 M0 비중을 높인다.
6. 특징의 효과 방향을 고정하지 않는다. 동일 특징이 상태에 따라 양수·음수·0이 될 수 있어야 한다.
7. 번호쌍 효과는 강하게 수축하고, 희소한 우연 패턴을 확대하지 않는다.
8. 군중 회피 영향은 전체 판단의 5%를 초과하지 않는다.

## 불확실성과 검증

9. 사용자 화면에는 95% 신용구간 또는 예측구간을 표시한다.
10. 99.9%는 당첨 정확도가 아니라 비균등모형 승격 기준이다.
11. 미래 데이터 누출을 허용하지 않는다.
12. 성공 사례뿐 아니라 모든 실패 결과를 보존한다.
13. 균등 기준 대비 우위가 없으면 반드시 `통계적 우위 없음`을 표시한다.
14. 백테스트 성능만으로 모델을 확정하지 않고 미래 walk-forward 재현을 요구한다.

## 데이터

15. 과거 당첨번호는 단일 원본에서만 읽는다.
16. `auto_checked` 데이터는 사용자 승인된 연구용 백테스트와 합성검증에만 사용할 수 있다. 실제 미래예측 공개와 데이터 잠금에는 `verified`·`locked` 데이터만 사용한다.
17. `verified` 및 `locked` 레코드를 덮어쓰지 않는다.
18. 정정 시 이전 값과 변경 이력을 보존한다.
19. HTML·JavaScript·테스트 코드에 당첨번호를 중복 하드코딩하지 않는다.

## 제품

20. 출력은 6개 번호 조합 5세트다.
21. 보너스번호는 저장·표시하지만 v1 예측 대상에서 제외한다.
22. `예측하기` 버튼은 브라우저에서 새 난수를 생성하지 않는다.
23. 동일 회차·동일 모델 버전에는 동일한 잠금 예측을 제공한다.
24. 상세 분석은 기본 숨김이며 사용자가 버튼으로 연다.
25. Supabase 연결은 Gate 4 승인 전 진행하지 않는다.

## 개발 운영

26. `main`에서 직접 개발하지 않는다.
27. 사용자 검토 전 PR을 임의 병합하지 않는다.
28. 수식 변경은 사용자 승인과 모델 버전 증가 없이 수행하지 않는다.
29. 모든 작업 종료 시 `handoff/PROJECT_HANDOFF.md`와 `handoff/WORK_LOG.md`를 갱신한다.
30. 새로운 의사결정은 `handoff/DECISION_LOG.md`에 누적한다.

위 원칙과 충돌하는 요청이나 구현을 발견하면 작업을 중단하고 사용자에게 설명해야 합니다.
