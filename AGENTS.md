# AGENTS.md

이 문서는 Codex 및 모든 AI 개발 에이전트가 **작업 시작 전에 반드시 읽어야 하는 최상위 운영 규칙**입니다.

## 1. 프로젝트 목적

이 프로젝트는 과거 로또 6/45 데이터와 이후 누적 데이터를 이용해 다음 회차의 6개 번호 조합 5세트를 생성하고, 해당 예측을 사전에 잠근 뒤 실제 결과와 비교하여 검증하는 연구형 확률예측 시스템입니다.

단순 고빈도 번호 추천기가 아닙니다. 핵심은 다음입니다.

- 균등 무작위 기준모형을 항상 유지
- 지속·반전·구조변화 가설을 동시에 평가
- 무작위성 검정을 비균등모형 활성화 게이트로 사용
- 불확실성과 실패 이력을 숨기지 않음
- 미래 데이터 누출 금지
- 동일 입력·동일 버전에서 동일 결과 재현

## 3. 브랜치 및 PR 원칙

- `main`에 직접 개발하지 않습니다.
- 작업별 `feature/*`, `fix/*`, `docs/*` 브랜치를 사용합니다.
- 작업 결과는 Draft PR로 제출합니다.
- 사용자 검토 전 임의 병합하지 않습니다.
- 이미 검증된 브랜치·태그·모델 버전을 덮어쓰지 않습니다.
- Gate 2-3 합성 calibration은 Gate 2-2 사용자 승인 전 시작하지 않습니다.

## 4. 알고리즘 변경 금지 원칙

`docs/ALGORITHM_SPEC.md`의 v1.0.0 수식은 현재 고정 명세입니다.

다음은 리팩터링 명목으로 변경할 수 없습니다.

- M0 균등 무작위모형 제거
- M1 지속, M2 반전, M3 구조변화 모형 중 하나 제거
- 동적 혼합분포를 단일 빈도 점수식으로 축소
- 무작위성 게이트를 추천점수로 변질
- 신호가 없을 때 균등모형으로 복귀하는 규칙 제거
- 95% 불확실성 표시 및 99.9% 모델 승격 기준 완화
- 미래 회차 결과를 특징·가중치·하이퍼파라미터 선택에 사용
- 정확히 6개를 선택하는 조합분포를 독립 Bernoulli 최종모형으로 대체
- pair interaction을 검증과 승인 없이 활성화

수식 변경이 필요하다고 판단하면 코드를 먼저 바꾸지 말고:

1. 변경 사유
2. 기존 수식과의 차이
3. 기대효과
4. 과적합 위험
5. 검증 방법
6. 모델 버전 변경안

을 문서로 작성하여 사용자 승인을 받아야 합니다.

## 5. 데이터 무결성과 사용범위

- 과거 당첨번호는 하나의 원본 데이터 파일 또는 향후 Supabase 원본 테이블에서만 읽습니다.
- HTML, JavaScript, 테스트 코드에 당첨번호를 중복 하드코딩하지 않습니다.
- `auto_checked` 데이터는 승인된 Gate 2 연구용 Walk-forward와 합성검증에만 사용할 수 있습니다.
- 공식 확인 전 데이터로 실제 미래예측을 공개하거나 잠그지 않습니다.
- `verified` 및 `locked` 데이터는 직접 덮어쓰지 않습니다.
- 정정은 변경 이력을 보존하는 방식으로만 수행합니다.
- 번호 범위·중복·회차·날짜·보너스번호·체크섬 검사를 통과해야 합니다.
- 모든 연구용 출력에는 `research_only: true`, `public_release_allowed: false`를 포함합니다.

## 6. 예측 버튼의 의미

HTML의 `예측하기` 버튼은 클릭할 때마다 새 난수를 생성하거나 브라우저에서 모델을 다시 학습시키는 버튼이 아닙니다.

- Python 엔진이 해당 회차의 예측을 사전에 계산
- 모델 버전·입력 데이터 범위·생성시각·해시와 함께 잠금
- 버튼은 잠금된 5세트를 사용자에게 공개
- 같은 회차·같은 모델 버전에서는 모든 사용자에게 동일한 결과 제공
- 공식 데이터 잠금 전 연구용 후보는 public production 화면에 노출하지 않음

## 7. 작업 종료 시 필수 업데이트

모든 작업 세션의 마지막에는 반드시 다음을 갱신합니다.

### `handoff/PROJECT_HANDOFF.md`

- 현재 상태
- 완료 항목
- 미완료 항목
- 다음 작업
- 주의사항
- 최신 브랜치와 PR

### `handoff/WORK_LOG.md`

- 작업일
- 작업자 또는 에이전트
- 변경 파일
- 수행 내용
- 검증 결과
- 남은 위험

### `handoff/DECISION_LOG.md`

새로운 의사결정이 발생한 경우 근거와 영향을 추가합니다. 기존 기록은 삭제하지 않습니다.

## 8. 비개발자 사용자 보고 방식

사용자는 비개발자입니다. 작업 완료 보고에는 다음을 명확히 적습니다.

- 무엇을 변경했는지
- 왜 변경했는지
- 현재 무엇이 작동하는지
- 검증을 통과했는지
- 사용자가 확인할 화면 또는 링크
- 다음 단계에서 사용자가 결정할 사항

내부 클래스명이나 함수명만 나열하지 마십시오.

## 9. Gate 운영

- Gate 0: 계획·수식·정책 승인 — 완료
- Gate 1: 데이터·아카이브 기반 — 사용자 승인 완료
- Gate 2-1: 예측 엔진 명세 — 사용자 승인 완료
- Gate 2-2: 엔진 골격·수학적 정합성 — 현재 구현 및 검증 단계
- Gate 2-3: 합성 null/positive control
- Gate 2-4: 역사적 Walk-forward
- Gate 2-5: 검증보고 및 연구용 후보
- Gate 3: HTML UX
- Gate 4: 수식과 코드 일치 감사 및 사용자 검수
- Gate 5: Supabase 이전과 자동화

현재 승인된 Gate를 넘어서는 작업은 사용자 승인 없이 진행하지 않습니다. 특히 Supabase 연결은 Gate 4 승인 전 진행하지 않습니다.
